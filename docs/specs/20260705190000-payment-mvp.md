# EP-6 - Payment MVP

Status: implemented.

## Problem and solution

Attendees with paid Pending orders need a secure way to complete payment without EventHub collecting card details, adding a platform fee, or holding organizer funds. Free orders must continue without a payment step.

The solution is a provider-led payment flow. EventHub initiates a provider checkout for non-zero Pending orders, stores only provider-safe payment references and status, confirms orders from provider success notifications, and treats duplicate success notifications as safe no-ops. Failed or abandoned payments leave the order Pending so the existing hold expiry flow can release inventory and the attendee can retry while the hold remains valid.

## Acceptance criteria

**AC-01:** Given a Pending order with a non-zero total, when the attendee starts payment, then EventHub creates or reuses a provider payment attempt and returns a provider redirect URL.

**AC-02:** EventHub never collects or stores card details; stored payment data is limited to order id, amount, currency, provider reference, status, and timestamps.

**AC-03:** Given a zero-total order, when it is placed, then it is Confirmed immediately without a payment step.

**AC-04:** Given the provider reports successful payment, when EventHub processes the notification, then the payment is Captured and the related order becomes Confirmed.

**AC-05:** Given the same successful provider notification is delivered more than once, when EventHub processes it again, then the order remains Confirmed and the payment is not captured twice.

**AC-06:** Given the provider reports payment failure, when EventHub processes the notification, then the payment is Failed, the order remains unconfirmed, and inventory stays reserved only until the normal hold expiry.

**AC-07:** Given the provider checkout is abandoned without a success notification, when the hold expires, then the order expires through the existing F-5.5 flow and inventory returns to availability.

**AC-08:** The amount sent to the provider equals the displayed order total; EventHub adds no platform fee and records no platform-fund settlement.

## Domain and business rules

- Payments belong to the Payments bounded context (`BC-4`) and settle Sales orders (`BC-3`) through an external provider.
- A payment amount must equal the order total and use the same configured currency.
- Only Pending non-zero orders can start provider payment.
- A zero-total order is confirmed by Sales without a Payment aggregate.
- Payment status transitions are limited to Initiated -> Captured, Initiated -> Failed, and Captured -> Refunded in future refund work.
- Provider references are idempotency keys at the EventHub boundary; duplicate success or failure notifications do not duplicate side effects.
- Confirmation of a non-zero order requires a captured payment reference.
- EventHub stores no card number, expiry, CVC, tokenized card profile, or payout-account balance.

## UI behavior or API contract

- `POST /api/orders/{orderId}/payments` starts payment for a Pending paid order and returns a redirect URL and provider reference.
- Provider success/failure notifications are accepted through server-side API endpoints. For the MVP no browser card form is hosted by EventHub.
- Checkout and order-status screens can launch provider checkout for paid Pending orders, redirecting the attendee away from EventHub for card entry and remaining retryable while the order is Pending.
- Existing order status reads reflect Pending, Confirmed, or Expired after payment or hold processing, including the provider-return case where confirmation has not arrived yet.
- Zero-total orders continue to return Confirmed from the existing order placement response.

## Data, real-time, security, edge cases, dependencies, assumptions, out of scope

**Data:** Store payment attempt status, order id, amount, currency, provider reference, and timestamps in PostgreSQL.

**Real-time:** No new realtime channel is required in this slice.

**Security:** Payment notification handling is provider-boundary logic and must not require or store card details. Signature verification can be provider-adapter-specific; the local no-op provider used for development is not a real money path.

**Edge cases:** Starting payment for a free, expired, cancelled, confirmed, missing, or amount-mismatched order is rejected. Duplicate provider callbacks are safe. Failed payment does not expire the order early.

**Dependencies:** F-5.3 creates Pending orders; F-5.5 expires abandoned holds; F-5.6 exposes order status.

**Assumptions:** The MVP provider adapter can be a deterministic local/no-op gateway suitable for development and integration tests while preserving the Application port boundary for a trusted provider.

**Out of scope:** Refunds on cancellation (F-6.6), ticket issuance (EP-7), real provider credentials, EventHub-hosted card collection UI, payout account onboarding, platform fees, subscriptions, and provider-specific checkout-page branding.

## Adjacent Feature Boundary

**Neighboring features:** F-5.3 creates Pending orders and reservations; F-5.5 expires abandoned holds; F-5.6 reads order status; EP-7 issues tickets from confirmed orders; F-6.6 refunds paid orders after event cancellation.

**In this slice:** F-6.1 through F-6.5 for MVP payment initiation, free-order confirmation, provider-driven success/failure handling, idempotency, and no-fee/no-card-data behavior.

**Out of scope:** F-6.6 refunds, ticket persistence/issuance, notification delivery, real provider credential setup, payout onboarding, and provider-specific payment page polish.
