# Feature: Guest checkout

> Features: F-5.2 | Status: IMPLEMENTED | Date: 2026-07-04
> PRD: DEC-3, QG-1, QG-4, QG-6 | DDD: BC-3 AGG-Order, VO-Contact | Tech: Sections 4, 7

## 1. Problem & Solution

**Problem:** After a buyer selects valid ticket quantities, checkout must stay low-friction. Requiring an attendee account would add avoidable friction and conflict with the MVP attendee journey.

**Solution:** Let the attendee continue as a guest by entering only a name and valid email address. The guest contact becomes the buyer contact on the order. Account creation, attendee wallets, payment capture, and ticket delivery remain separate feature slices.

**Personas:** PER-A1 needs a fast purchase path. PER-A2 needs to buy for a group without creating an account first.

**Scope:** F-5.2 only. This slice collects and validates guest contact details and submits them with the validated ticket selection. It may use the existing order-placement endpoint where present, but it does not add account creation, payment provider collection, ticket issuance, delivery, order status recovery, or attendee wallet behavior.

## 2. Acceptance Criteria

**AC-01:** GIVEN a buyer has started checkout with a valid ticket selection, WHEN they view the checkout step, THEN they can enter a name and email without signing in.

**AC-02:** GIVEN the buyer provides a non-empty name and a syntactically valid email, WHEN they continue, THEN the purchase request includes that contact and no account creation is required.

**AC-03:** GIVEN the buyer omits the name, WHEN they submit, THEN checkout is blocked with a clear validation message.

**AC-04:** GIVEN the buyer provides an invalid email, WHEN they submit, THEN checkout is blocked with a clear validation message.

**AC-05:** GIVEN the buyer is anonymous, WHEN the contact and ticket selection are valid, THEN the request can complete through the public checkout path.

**AC-06:** GIVEN the ticket selection becomes invalid while the buyer enters contact details, WHEN they submit, THEN checkout reports the current selection problem and does not require account creation.

**AC-07:** GIVEN the buyer uses a phone-sized viewport, WHEN they enter guest details and review the order summary, THEN the form remains usable without horizontal scrolling or clipped controls.

## 3. Domain & Business Rules

- A guest attendee is represented by contact details: name plus email.
- Name is required for the buyer contact.
- Email is required and must be well-formed.
- Guest checkout must not require a user account or organizer role.
- Personal data collection stays minimal for this slice: only buyer name and email.
- Ticket selection and inventory validation remain authoritative in the checkout/order path from F-5.1 and F-5.3.

## 4. UI Behavior or API Contract

- The checkout page shows the validated ticket selection and a guest contact form.
- The primary action submits name, email, and selected ticket lines.
- Validation errors appear next to or near the form fields and are written in attendee-readable language.
- Server-side validation errors are shown without exposing implementation details.
- On success, the attendee sees that the order was accepted and can identify the created order.
- API requests use contract DTOs and must not expose domain entities.

## 5. Data, Real-Time, Security, Edge Cases, Dependencies, Assumptions, Out of Scope

**Data:** The buyer contact is stored with the order when order placement occurs. No account record is created.

**Real-time:** No real-time behavior is required.

**Security and privacy:** The endpoint remains public for attendees. Only required contact data is collected. Payment data is not collected in this slice.

**Edge cases:** Empty or whitespace names fail validation. Invalid email syntax fails validation. Stale ticket selections return the current checkout/order validation error. Anonymous users must not be redirected to sign in.

**Dependencies:** F-5.1 provides the ticket selection entering checkout. Existing order-placement behavior may create the pending order and hold, but that remains owned by F-5.3.

**Assumptions:** A single buyer contact applies to all lines in the order for MVP.

**Out of scope:** Attendee account creation, email verification, payment collection, ticket issuance, ticket delivery, order status pages, attendee wallet, and marketing opt-ins.

## Adjacent Feature Boundary

**In scope for F-5.2:** guest contact collection, client and server validation, anonymous checkout submission using selected ticket lines.

**Neighboring features:** F-5.1 selects tickets and starts checkout. F-5.3 creates the pending order and inventory hold. F-5.4 shows final price summary. EP-6 handles payment. EP-7 handles ticket issuance and delivery.

**Out of scope:** durable order semantics beyond the existing placement endpoint, hold expiry, payment, tickets, attendee accounts, and organizer reporting.
